# pptx-to-clean-templates

**Диагностирует любую презентацию и превращает «ручные» слайды в настоящие переиспользуемые макеты — автоматически.**

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)
![Python](https://img.shields.io/badge/python-3.12%2B-blue.svg)

[English](README.md) · **Русский**

AI-генераторы презентаций умеют применить *бренд-кит* — ваши цвета, шрифты и логотип — но не
могут воспроизвести реальные макеты **вашего** корпоративного шаблона. А большинство реальных
корпоративных дек ещё хуже: они собраны руками, плавающими текст-боксами и фоновыми картинками
во весь слайд, **без единого настоящего макета**. Наведите на такой файл любой инструмент
«сгенерируй по моему шаблону» — получите безликий пустой результат.

`pptx-to-clean-templates` чинит вход. Он:

1. **Аудирует** любой `.pptx`/`.potx` и человеческим языком объясняет: пригоден ли файл, что в нём
   собрано вручную и насколько это лечится автоматически.
2. **Лечит** — находит повторяющиеся *типы* слайдов (обложка, карточка дня, страница отеля…) и
   переписывает их в настоящие макеты PowerPoint (`p:sldLayout` на мастере), перенося шрифты,
   цвета, фоны и типографику уровней абзаца так, что новые слайды наследуют фирменный стиль
   **по построению**.

На выходе — обычный `.pptx`, который открывается в PowerPoint, или подаётся в любой генератор,
заполняющий плейсхолдеры.

## Чем отличается

Большинство инструментов останавливаются на **бренд-ките** (цвета + шрифты + логотип). Этот
воссоздаёт саму **структуру макетов** из файла, где её могло не быть вовсе, — та самая непарадная
OOXML-хирургия, которую пропускают инструменты «примени мой бренд».

## Установка

```bash
pip install git+https://github.com/620x/pptx-to-clean-templates
```

Для разработки:

```bash
git clone https://github.com/620x/pptx-to-clean-templates
cd pptx-to-clean-templates
uv sync            # или: pip install -e ".[dev]"
```

Нужен Python 3.12+.

## CLI

```bash
# Диагностика шаблона
x6 audit deck.pptx
x6 audit deck.pptx --json report.json

# Лечение «ручных» слайдов в настоящие макеты
x6 heal deck.pptx -o healed.pptx --probe preview.pptx
```

`--probe` пишет пробу (по одному слайду на макет), чтобы увидеть восстановленные макеты до какой-либо генерации.

## Библиотека

```python
from x6pptxgen.template.inventory import build_inventory
from x6pptxgen.audit.checks import run_checks
from x6pptxgen.heal.engine import heal_template, build_probe

inv = build_inventory("deck.pptx")          # структурный инвентарь шаблона (сериализуется в JSON)
findings = run_checks(inv)                    # находки-антипаттерны, человеческим языком

report = heal_template("deck.pptx", "healed.pptx")
build_probe("healed.pptx", "preview.pptx")    # по слайду на каждый новый макет
```

## Как работает лечение

1. **Кластеризация** слайдов-примеров по структурной сигнатуре — копипаст-вёрстка даёт
   повторяющиеся семейства, и эти семейства становятся макетами.
2. **Выбор донора** в каждом семействе (самый полный пример) и **разметка ролей** его текст-зон
   по кеглю, позиции и длине — заголовок, кикер, подзаголовок, тело, примечание — никогда по
   содержимому.
3. **Пересборка** семейства в настоящий макет: статичный декор копируется (с перепривязкой
   image/hyperlink-связей), контентные зоны становятся плейсхолдерами с типографикой, перенесённой
   в `lstStyle` (наследуется по построению), фон донорского слайда пересаживается в макет, а
   таблицы становятся колонками-плейсхолдерами. Когда свободные макеты кончаются — создаются
   новые layout-части на уровне OPC.

## Честные ограничения

- Разметка ролей — **эвристика**: выверена на реальных деках, но не идеальна на каждом файле.
  Проба существует ровно для того, чтобы посмотреть результат глазами.
- **Замер текста / защита от переполнения здесь не решены.** `measure/` пока только находит файлы
  шрифтов; точный замер ёмкости (через fontTools) — задел на будущее.
- Эвристика доступности шрифтов смотрит в **папки шрифтов macOS**; на других ОС недооценивает
  (на само лечение не влияет).
- Находки аудита и prompt'ы плейсхолдеров сейчас на **русском** (происхождение движка). **i18n —
  отличный первый вклад**: строки живут в `audit/checks.py`, `audit/report.py`, `heal/engine.py`.

## Область

Это движок, вырезанный из более крупного проекта. Генерация контента ИИ, конвейер брифа и
web-редактор сюда сознательно **не** входят — этот репозиторий про анализ и лечение шаблона,
и он самодостаточен.
